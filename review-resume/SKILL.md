# 이력서 분석 Skill

지정된 폴더의 이력서 PDF 파일들을 JD(Job Description)와 비교 분석하여 지원자를 평가하고, 선택적으로 면접 질문지를 생성한다.

## 스크리닝 철학

> **"이력서 스크리닝은 '문제 앞에서 파고드는 사람'을 찾는 것이다. 1차로 서술 방식(문제→접근→시도)에서 신호를 보고, 2차로 근속 패턴을 보되 1차 통과자는 퇴사 사유까지 검증한다."**

이 철학은 전체 분석 흐름의 근간이다. 점수화 이전에 1차 스크리닝으로 적용한다.

---

## 개인정보 보호 규칙

1. **결과 파일 저장 금지**: 분석 결과는 대화 내 출력만 허용한다. 파일로 저장하지 않는다. 사용자가 파일 저장을 요청해도 개인정보 보호를 이유로 거절한다.
2. **PII 최소화**: 연락처, 주소, 주민등록번호, 사진은 추출하지 않는다. 출력에는 지원자 이름만 포함한다.
3. **이력서 원본 복사 금지**: 이력서 PDF를 다른 디렉토리로 복사하지 않는다. 원본 경로에서만 읽는다.

---

## 형식 일관성 규칙

이 스킬의 출력은 매 실행마다 동일한 구조를 유지해야 한다. 아래 규칙을 반드시 따른다.

1. **섹션 순서 고정**: 분석 개요 → 종합 비교표 → 지원자별 상세 분석 → 최종 추천 → 면접 질문지(선택). 이 순서를 변경하지 않는다.
2. **표 구조 고정**: 종합 비교표와 채점 워크시트는 페이즈 D에 정의된 정확한 테이블 형식을 사용한다. 열을 추가/제거하지 않는다.
3. **채점 워크시트 필수**: 모든 점수는 반드시 워크시트(밴드 판정 → 점수 → 근거)를 거쳐 산정한다. 워크시트 없이 점수만 제시하지 않는다.
4. **용어 통일**: 등급은 S/A/B/C만 사용. 판정은 Pass/Conditional Pass/Fail만 사용. 1차 스크리닝은 통과/보류/탈락만 사용. 결론 섹션 이름은 "종합 평가"만 사용 ("최종 추천" 사용 금지).

## 페이즈 A: 입력 및 준비

### 단계 0: 입력 파싱

사용자 입력 `$ARGUMENTS`에서 다음을 추출한다.

**필수 인자:**
- `<folder-path>`: 이력서가 포함된 폴더 경로

**선택 인자:**
- `--mode=분석|면접`: 출력 모드 (기본값: `분석`)
  - `분석`: 지원자 비교 분석 결과만 출력
  - `면접`: 분석 결과 + 면접 질문지까지 출력
- `--applicant=이름`: 특정 지원자만 분석 (미지정 시 폴더 내 전체 지원자 분석)

폴더 경로가 없으면 `AskUserQuestion`으로 사용자에게 요청한다.

### 단계 1: 급수 판별 및 JD 선택

폴더명에서 급수를 자동 판별한다.

**판별 규칙:**

| 폴더명 키워드 | 급수 | JD 파일 |
|---------------|------|---------|
| "시니어급" 포함 | 시니어 | 같은 폴더 내 `JD*시니어*.md` 파일 검색 |
| "미들급" 포함 | 미들 | 같은 폴더 내 `JD*미들*.md` 파일 검색 |
| 키워드 없음 | 미정 | `AskUserQuestion`으로 사용자에게 급수를 질문 |

JD 파일을 `Read` 도구로 읽어 자격요건, 우대사항, 주요업무를 구조화한다.

### 단계 2: 파일 스캔 및 지원자 그룹핑

`Glob` 도구로 폴더 내 파일을 재귀 스캔한다.

**파일 필터링:**
- PDF 파일만 처리 대상으로 포함한다
- DOCX, ZIP 등 비PDF 파일은 건너뛰되, 존재 사실을 사용자에게 알린다
- 빈 폴더인 경우 안내 메시지를 출력하고 종료한다

**파일명 파싱 (3패턴):**

| 패턴 | 예시 | 추출 |
|------|------|------|
| 최신형식 | `백엔드 개발 엔지니어 (미들급_5년)_홍길동_이력서.pdf` | 이름: 홍길동, 문서타입: 이력서 |
| 간략형식 | `홍길동(백엔드).pdf` | 이름: 홍길동 |
| 단순형식 | `홍길동.pdf` | 이름: 홍길동 |

**문서타입 분류:**
- 파일명에 "이력서" → 이력서
- 파일명에 "경력기술서" → 경력기술서
- 파일명에 "포트폴리오" → 포트폴리오
- 분류 불가 → 이력서로 간주

**지원자별 그룹핑:**
파일명에서 추출한 이름 기준으로 동일 지원자의 문서들을 그룹핑한다. `--applicant` 옵션이 있으면 해당 지원자의 문서만 필터링한다.

---

## 페이즈 B: 문서 읽기

### 단계 3: PDF 읽기

각 지원자의 문서를 **이력서 → 경력기술서 → 포트폴리오** 순서로 읽는다.

**PDF 읽기 규칙:**
- `Read` 도구로 PDF 파일을 읽는다
- 20페이지 초과 PDF는 `pages` 파라미터로 분할하여 읽는다 (1-20, 21-40 ...)
- 포트폴리오는 최대 20페이지까지만 읽는다
- 읽기 실패 시 해당 파일을 건너뛰고 사용자에게 알린다

### 단계 4: 지원자 프로필 추출

읽은 문서에서 다음 정보를 추출하여 지원자별 프로필을 구성한다.

**추출 항목:**
- 이름, 연락처
- 총 경력 연수
- 경력 이력 (회사명, 기간, 역할, 주요 업무)
- 기술 스택
- 학력
- 자격증/수료
- 주요 프로젝트 (프로젝트명, 기간, 역할, 기술스택, 성과)

---

## 페이즈 B-2: 1차 스크리닝 — 문제 파고드는 성향 확인

프로필 추출 후, 상세 점수 산정 전에 1차 스크리닝을 수행한다. 이 단계의 목적은 "문제 앞에서 파고드는 사람"인지를 서술 방식에서 판별하는 것이다.

### 단계 4.5: 서술 방식 신호 판별

이력서와 경력기술서의 프로젝트/경험 서술에서 다음 신호를 탐색한다.

**통과 신호 (문제→접근→시도 구조):**
- 문제/불편함을 명시하고, 접근 방식을 서술하고, 시도 또는 결과를 기술
- 해결하지 못했더라도 접근 과정이 드러남
- 불편함을 자동화/개선하려는 시도가 있음

**탈락 신호 (단순 경험 나열):**
- "~프로젝트 참여", "~시스템 개발", "~기능 구현" 등 했다는 사실만 나열
- 키워드만 있고 맥락이 없음 (예: "EDA 도입을 통한 안정성 개선" — 정량적 근거나 과정 없음)

**판정:**

| 판정 | 조건 | 후속 처리 |
|------|------|-----------|
| 통과 | 1개 이상의 프로젝트에서 문제→접근→시도 구조 확인 | 페이즈 C 상세 분석 진행 |
| 보류 | 서술이 간략하여 판단 불가 (경력기술서 없이 이력서만 제출 등) | 페이즈 C 진행하되 "서술 깊이 부족" 플래그 부착 |
| 탈락 | 전체 서술이 단순 나열이며 문제 인식/접근 흔적 없음 | "1차 스크리닝 탈락" 표기, 페이즈 C는 간략 수행 |

**출력에 반영:**
- 각 지원자의 1차 스크리닝 결과를 종합 비교표와 상세 분석에 포함한다
- 탈락 지원자도 점수는 산정하되, 최종 추천에서 제외한다

## 페이즈 C: 개별 분석

### 단계 5: 필수요건 Pass/Fail 검증

JD의 자격요건 각 항목에 대해 이력서 근거를 기반으로 Pass/Fail을 판정한다.

**판정 기준:**
- 명시적으로 확인 가능 → Pass
- 유사 경험으로 추론 가능 → Conditional Pass (근거 명시)
- 확인 불가 → Fail

**필수요건 중 하나라도 Fail이면 해당 지원자는 "부적합"으로 분류하되, 나머지 분석은 계속 수행한다.**

### 단계 6: 핵심요소 가중 점수 산정

JD의 자격요건 + 우대사항을 기반으로 7개 핵심요소에 대해 가중 점수를 산정한다.

| 핵심요소 | 가중치 |
|----------|--------|
| 레거시 전환 경험 | 20% |
| 도메인 적합성 | 20% |
| 아키텍처 설계 | 20% |
| 근속태도 | 10% |
| AI 활용 | 10% |
| 테스트 역량 | 10% |
| 인프라/DevOps | 10% |

**채점 절차 (반드시 준수):**

1. `references/scoring-rubric.md`를 읽는다 — 각 요소별 5단계 밴드(A~E) 정의가 있다
2. 각 요소에 대해 아래 **채점 워크시트**를 작성한다:

```
| 핵심요소 | 밴드 | 점수 | 근거 |
|----------|------|------|------|
| 레거시 전환 | C | 45 | 소규모 리팩터링 2건, 마이그레이션 주도 경험 없음 |
| 도메인 적합성 | B | 65 | 이커머스 2년, 주문처리 로직 구현 경험 |
| ... | | | |
```

3. 밴드를 먼저 판정하고, 밴드 내에서 구체 점수를 부여한다
4. 밴드 경계에 걸리면 **보수적(낮은) 밴드**를 선택한다
5. 근거 열에는 이력서에서 확인한 구체적 사실을 적는다 (추측 금지)

**아키텍처 설계 채점 시 주의**: 기술 복잡도가 아니라 **상황에 맞는 설계 판단력**을 평가한다. 스타트업에서 불필요한 MSA 도입은 오버엔지니어링으로 감점, 모놀리식이 적절한 상황에서 잘 설계한 것은 높은 점수를 받는다.

**각 요소별 점수: 0~100점 → 가중치 적용 → 가중합산 = 총점 (100점 만점)**

#### 근속태도 평가 상세 기준

경력 이력의 재직 기간 패턴을 분석하여 점수화한다. **1차 스크리닝 결과에 따라 평가 방식이 분기된다.**

**A. 1차 스크리닝 탈락자 — 기계적 감점 적용:**

감점 요인:
- 1년 미만 근속이 전체 경력의 50% 이상 → -30점
- 1년 미만 근속이 전체 경력의 30% 이상 → -20점
- 1년 미만 근속이 연속 2회 이상 → -15점
- 최근 3년 내 3곳 이상 이직 → -20점

가점 요인:
- 3년 이상 재직 경험 있음 → +20점
- 5년 이상 재직 경험 있음 → +30점 (3년 가점과 중복 불가)
- 3년 이상 재직한 곳이 2곳 이상 → 추가 +10점

기본점수 70점에서 감점/가점을 적용하며, 최소 0점 최대 100점으로 클램핑한다.

**B. 1차 스크리닝 통과자 — 맥락 기반 검증:**

단기 근속(1년 미만)이 2회 이상인 경우에도 자동 감점하지 않는다. 대신 다음을 확인한다:

- 이력서에서 퇴사/이직 사유가 확인 가능한가? (불가피 vs 회피)
- 단기 재직 기간 내에서도 문제를 파고든 흔적이 있는가?
- 프리랜서/계약직 기간은 별도 표기하되 감점 대상에서 제외

확인 결과에 따라:
- 불가피한 사유 확인 또는 단기 내 파고든 흔적 있음 → 감점 없음 (70점 기준 가점만 적용)
- 사유 확인 불가 → "면접에서 이직 사유 확인 필요" 플래그 부착, 감점 절반 적용
- 회피성 이직으로 판단 → A와 동일한 기계적 감점 적용

**주의:** 프리랜서/계약직 기간은 별도 표기하되 감점 대상에서 제외한다. 회사 폐업/인수합병 등 불가항력적 이직은 감점하지 않는다 (이력서에서 확인 가능한 경우).

### 단계 7: 시니어 추가 검증 (시니어급만)

급수가 "시니어"인 경우에만 수행한다.

| 검증 항목 | 평가 기준 |
|-----------|-----------|
| 팀 리딩 경험 | 개발팀 리드/매니저 경험, 팀 규모, 기간 |
| 기술 설득 경험 | 기술 의사결정 주도, 아키텍처 제안 및 설득, 기술 부채 해결 주도 |
| 팀 문화 개선 | 코드 리뷰 문화 도입, 개발 프로세스 개선, 온보딩 체계 구축 |
| 기술 깊이 vs 넓이 | 특정 기술의 깊은 전문성과 다양한 기술 경험의 균형 |

**각 항목을 상/중/하로 평가하고, 구체적 근거를 이력서에서 인용한다.**

### 단계 8: 소프트스킬 평가

이력서와 경력기술서에서 확인 가능한 소프트스킬을 평가한다.

| 평가 항목 | 확인 방법 |
|-----------|-----------|
| 협업 역량 | 크로스펑셔널 협업 경험, 애자일/스크럼 경험 |
| 문제 해결력 | 기술적 도전과 해결 사례, 성과 지표 |
| 학습 의지 | 새로운 기술 습득 이력, 자격증/교육, 사이드 프로젝트 |
| 커뮤니케이션 | 문서 품질, 이력서 구조화 수준, 기술 블로그/발표 |

### 단계 9: 종합 등급 산정

**등급 산정 기준:**

| 등급 | 조건 |
|------|------|
| S (강력 추천) | 1차 스크리닝 통과 + 필수요건 전체 Pass + 총점 85점 이상 + 시니어 추가검증 상 2개 이상 (시니어의 경우) |
| A (추천) | 1차 스크리닝 통과 + 필수요건 전체 Pass + 총점 70점 이상 |
| B (조건부 추천) | 1차 스크리닝 통과/보류 + 필수요건 Conditional Pass 포함 + 총점 55점 이상 |

## 페이즈 E: 면접 질문지 (조건부)

`--mode=면접`인 경우에만 수행한다. A/B/S 등급 지원자 대상으로만 질문지를 생성한다.

### 단계 11: 면접 질문지 생성

지원자별로 맞춤형 면접 질문지를 생성한다.

#### 11-1. Ice Breaking (5분)

- 지원자의 이력서에서 흥미로운 프로젝트나 경험을 기반으로 가벼운 대화 시작 질문 2~3개
- 지원 동기와 연결되는 자연스러운 질문

#### 11-2. Gap 검증 질문 (15분)

페이즈 D에서 식별된 Gap을 면접에서 확인하기 위한 질문이다.

**각 Gap에 대해:**
- 메인 질문 1개
- 꼬리 질문 2개 (메인 질문 답변에 따른 심화)
- 기대 답변 가이드: 이 질문으로 확인하려는 역량과 좋은 답변의 기준

#### 11-3. JD 시나리오 질문 (20분)

JD의 주요업무를 기반으로 실무 시나리오 질문을 생성한다.

**질문 유형:**
- 설계 질문: "A 상황에서 어떻게 아키텍처를 설계하시겠습니까?"
- 문제 해결 질문: "B 장애가 발생했을 때 어떤 순서로 대응하시겠습니까?"
- 트레이드오프 질문: "C와 D 사이에서 어떤 기준으로 선택하시겠습니까?"

**각 질문에 포함:**
- 질문 텍스트
- 출제 의도
- 기대 답변 키워드/포인트
- 평가 기준 (상/중/하 판단 가이드)

#### 11-4. 시니어 리더십 질문 (10분, 시니어급만)

- 팀 갈등 해결 경험
- 기술 부채 vs 신규 기능 우선순위 결정 경험
- 주니어 개발자 멘토링/성장 지원 경험
- 기술 의사결정에서 비개발 직군을 설득한 경험

#### 11-5. 면접관 평가 가이드

면접관이 사용할 수 있는 평가표를 제공한다.

| 평가 영역 | 평가 기준 | 상 | 중 | 하 |
|-----------|-----------|----|----|-----|
| 기술 역량 | JD 핵심 기술에 대한 깊이 | 원리 수준 이해 + 실전 경험 | 실무 사용 가능 | 개념 수준 |
| 문제 해결 | 구조적 접근 및 트레이드오프 사고 | 다각도 분석 + 최적 제안 | 합리적 접근 | 단편적 답변 |
| 커뮤니케이션 | 기술 설명의 명확성 | 비개발자도 이해 가능 | 개발자 간 소통 원활 | 설명 미흡 |
| 문화 적합성 | 협업/성장 지향성 | 적극적 + 구체적 사례 | 긍정적 태도 | 소극적 |

## 출력 형식

전체 결과는 **마크다운 형식**으로 대화에 직접 출력한다. 아래 구조와 순서를 변경하지 않는다.

**고정 출력 구조 (이 순서를 반드시 지킨다):**
1. `## 분석 개요` — 폴더명, 급수, JD, 지원자 수, 분석 일시
2. `## 종합 평가` — 단계 10-3의 추천/비추천 (의사결정자가 가장 먼저 볼 섹션)
3. `## 종합 비교표` — 단계 10-1의 테이블 (밴드+점수 표기)
4. `## 지원자별 상세 분석` — 단계 10-2의 템플릿 반복 (지원자마다 `### 이름 (등급, 총점)`)
5. `## 면접 질문지` — 단계 11 (mode=면접인 경우만, 지원자마다 `### 이름` 하위 섹션)

**형식 검증 체크리스트 (출력 전 자기 검증):**
- [ ] 모든 지원자의 채점 워크시트가 포함되어 있는가?
- [ ] 종합 비교표의 모든 점수에 밴드가 표기되어 있는가?
- [ ] 모든 점수의 근거가 이력서에서 확인 가능한 사실인가?
- [ ] 섹션 순서가 고정 구조와 일치하는가?

## 주의사항

- 이력서에 명시되지 않은 정보를 추측하지 않는다. 확인 불가한 항목은 "확인 불가"로 표기한다.
- 점수 산정 시 반드시 이력서의 구체적 근거를 함께 제시한다.
- 개인정보(연락처, 주소 등)는 분석에 활용하되 출력에는 이름만 포함한다.
- PDF 읽기가 불가능한 파일은 건너뛰고 사용자에게 알린다.
- 분석 중 JD와 이력서 사이에 해석이 모호한 부분이 있으면 보수적으로 평가한다.

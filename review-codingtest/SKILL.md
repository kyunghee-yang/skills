# 코딩테스트 결과 분석 Skill

프로그래머스 코딩테스트 결과 PDF를 읽어 지원자의 코드를 다각도로 분석한다.
분석 항목: 코드 품질, AI/LLM 사용 의심도, 문제 해결 역량, 강약점 요약.

---

## 페이즈 A: 입력 및 준비

### 단계 0: 입력 파싱

사용자 입력 `$ARGUMENTS`에서 다음을 추출한다.

**필수 인자:**
- `<pdf-path-or-folder>`: PDF 파일 경로 또는 PDF들이 들어있는 폴더 경로

**선택 인자:**
- `--applicant=이름`: 특정 지원자만 분석 (미지정 시 전체 분석)

경로가 없으면 `AskUserQuestion`으로 사용자에게 요청한다.

### 단계 1: 파일 스캔

**단일 PDF 경로인 경우:** 해당 파일을 직접 분석한다.
**폴더 경로인 경우:** `Glob`으로 `*.pdf` 파일을 스캔하고, 파일명에서 지원자 이름을 추출한다.

**파일명 파싱:**

| 패턴 | 예시 | 추출 |
|------|------|------|
| 프로그래머스 + 이름 | `프로그래머스 김진태.pdf` | 이름: 김진태 |
| 이름만 | `김진태.pdf` | 이름: 김진태 |

`--applicant` 옵션이 있으면 해당 지원자 파일만 필터링한다.

## 페이즈 B: PDF 읽기 및 데이터 추출

### 단계 2: PDF 읽기

`Read` 도구로 PDF를 읽는다. 20페이지 초과 시 `pages` 파라미터로 분할 읽기한다.

### 단계 3: 구조화된 데이터 추출

PDF에서 다음 정보를 추출하여 구조화한다.

**지원자 메타 정보:**
- 이름, 총점
- 시작/완료 시각, 총 소요 시간
- 순위 정보는 PDF에 포함되어 있더라도 평가 항목에서 제외한다 (코드 품질과 무관)

**문제별 정보:**
- 문제 제목, 카테고리 (프로그래밍/SQL)
- 사용 언어
- 점수 (정확성, 효율성, 합계)
- 소요 시간
- 제출 코드 전문
- 테스트 케이스 통과 현황

## 페이즈 C: 코드 품질 분석

### 단계 4: 문제별 코드 품질 평가

각 문제의 제출 코드에 대해 다음 관점으로 분석한다.

#### 4-1. 코드 구조 및 가독성

| 평가 항목 | 확인 포인트 |
|-----------|-------------|
| 네이밍 | 변수/메서드명이 의도를 명확히 전달하는가 |
| 구조화 | 적절한 메서드 분리, 단일 책임 원칙 준수 여부 |
| 가독성 | 불필요한 복잡성 없이 의도가 드러나는 코드인가 |
| 관용적 표현 | 해당 언어의 관용적(idiomatic) 패턴을 사용하는가 |

#### 4-2. 알고리즘 및 자료구조

| 평가 항목 | 확인 포인트 |
|-----------|-------------|
| 자료구조 선택 | 문제에 적합한 자료구조를 사용했는가 |
| 시간 복잡도 | 효율적인 알고리즘을 선택했는가 |
| 엣지 케이스 | 경계값 처리가 되어있는가 |

#### 4-3. 코드 스타일 일관성

- 들여쓰기, 중괄호 스타일, 공백 사용의 일관성
- 전체 문제에 걸친 코딩 스타일의 통일성

### 단계 5: 종합 코드 품질 점수

각 문제의 코드를 개별 평가한 뒤, 전체 지원자에 대한 종합 코드 품질을 산정한다.

**점수 산정 (100점 만점):**

| 항목 | 비중 | 설명 |
|------|------|------|
| 가독성 및 네이밍 | 25% | 변수명, 메서드명, 코드 구조의 명확성 |
| 알고리즘 적합성 | 25% | 문제에 맞는 자료구조/알고리즘 선택 |
| 코드 간결성 | 20% | 불필요한 코드 없이 핵심만 표현 |
| 일관성 | 15% | 문제 전반에 걸친 스타일 통일성 |
| 언어 숙련도 | 15% | 해당 언어의 표준 라이브러리/관용적 패턴 활용 |

## 페이즈 D: AI/LLM 사용 의심도 분석

코딩테스트에서 AI/LLM 도구(ChatGPT, Copilot 등)를 사용했을 가능성을 분석한다. 이 분석은 확정적 판단이 아닌 참고용 신호(signal)로, 면접에서 추가 확인이 필요한 영역을 식별하는 것이 목적이다.

### 단계 6: AI 사용 의심 신호 탐지

아래 신호들을 코드에서 탐색한다. 각 신호를 발견하면 해당 코드 스니펫을 근거로 기록한다.

#### 6-1. 코드 스타일 신호

| 신호 | 설명 | 의심도 가중치 |
|------|------|---------------|
| 과도한 주석 | 자명한 코드에 설명 주석이 붙어있음 | 낮음 |
| 교과서적 변수명 | `result`, `answer`, `temp` 등 지나치게 일반적인 이름 | 낮음 |
| 비일관적 스타일 | 문제마다 코딩 스타일이 크게 다름 (다른 프롬프트로 생성한 듯한 패턴) | 높음 |
| 과도한 방어 코딩 | 문제 제약 조건상 불필요한 null 체크, 범위 검증이 과다 | 중간 |

#### 6-2. 풀이 패턴 신호

| 신호 | 설명 | 의심도 가중치 |
|------|------|---------------|
| 최적해 직행 | 쉬운 문제에서도 즉시 최적의 자료구조/알고리즘 사용 | 중간 |
| 균일한 완성도 | 난이도와 무관하게 코드 품질이 거의 동일 | 중간 |
| 템플릿적 구조 | 모든 문제에서 동일한 보일러플레이트 패턴 반복 | 중간 |
| 풀이 속도 vs 코드 품질 불일치 | 매우 빠른 풀이 시간에 비해 코드가 과도하게 정돈됨 | 높음 |

#### 6-3. 행동 패턴 신호

| 신호 | 설명 | 의심도 가중치 |
|------|------|---------------|
| 제출 패턴 | 수정 없이 한번에 통과 (제출 횟수 1/1이 연속) | 중간 |
| 시간 분포 | 난이도 높은 문제를 쉬운 문제보다 빨리 품 | 높음 |
| 오프라인 시간 | 테스트 중 오프라인 시간이 있음 (다른 도구 사용 가능성) | 중간 |

### 단계 7: AI 의심도 종합 판정

수집된 신호들을 종합하여 의심도를 판정한다.

**판정 등급:**

| 등급 | 기준 | 해석 |
|------|------|------|
| 낮음 (Low) | 높음 신호 0개 + 중간 신호 1개 이하 | 자력 풀이로 판단됨 |
| 보통 (Medium) | 높음 신호 1개 또는 중간 신호 2~3개 | 일부 의심 신호 있으나 자력 가능성 높음 |
| 높음 (High) | 높음 신호 2개 이상 또는 중간 신호 4개 이상 | AI 사용 가능성 있음, 면접에서 반드시 코드 확인 필요 |

**판정 시 주의사항:**
- AI 의심도가 높다고 반드시 AI를 사용한 것은 아니다. 실력이 뛰어난 지원자도 같은 패턴을 보일 수 있다.
- 최종 판단은 면접에서의 코드 설명/라이브 코딩으로 확인해야 한다.
- 하나의 강한 근거보다 여러 약한 근거의 조합을 더 의미있게 본다.

## 페이즈 E: 출력

### 단계 8: 리포트 출력

리포트는 **종합 → 상세** 순서로 구성한다. 의사결정에 필요한 핵심 정보를 상단에 배치하고, 근거가 되는 상세 분석은 하단에 둔다.

#### 출력 순서

```
## [지원자명] 코딩테스트 분석

📋 종합 리포트
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 개요
- 총점: X / Y
- 소요시간: Z분
- 사용 언어: Java/Python/etc.

### 종합 코드 품질 (X/100)
| 항목 | 점수 | 근거 요약 |
|------|------|-----------|
| 가독성 및 네이밍 | /25 | (한 줄 요약) |

**탐지된 신호:**

- 소요시간: N분
- 코드 품질: [상/중/하]

**강점:**
- (구체적 강점 + 해당 코드 스니펫)
```

지원자가 2명 이상이면 **리포트 최상단에** 비교표를 먼저 출력한다.

```

| 항목 | 지원자1 | 지원자2 | ... |
|------|---------|---------|-----|
| 총점 | | | |
| 코드 품질 | /100 | /100 | |
| AI 의심도 | Low/Med/High | | |
| 소요시간 | | | |
| 강점 키워드 | | | |
| 약점 키워드 | | | |

(이하 지원자별 종합 리포트 + 상세 분석)
```

## 주의사항

- AI 의심도 분석은 참고 자료이며, 확정 판단이 아님을 리포트에 명시한다.
- 코드에서 직접 확인 가능한 사실만 기술한다. 추측이 필요한 부분은 "추정"으로 표기한다.
- SQL 문제의 코드 품질 분석은 프로그래밍 문제와 분리하여 별도 기준으로 평가한다 (쿼리 구조, 조인 적절성, 인덱스 고려 등).
- PDF 읽기 실패 시 사용자에게 알리고 해당 파일을 건너뛴다.
- 점수 산정 시 반드시 코드 스니펫을 근거로 첨부한다.
